Real-Time Anomaly Detection Pipeline

End-to-end streaming architecture for financial transaction fraud detection with online ML, drift detection, and auto-retraining

Kafka Streaming River Online ML Faust DSP Celery + TimescaleDB 6 Microservices Prometheus + Grafana
Architecture Overview
Pipeline Stages (hover for details)
🏭
Producer
:8001
kafka-python
Generates synthetic financial events at configurable rate. Cycles Normal → Drift → Attack phases.

Producer

  • Emits 100 events/sec to raw-transactions
  • 3 phases: Normal (0–99k), Drift (100k–199k), Attack (200k–249k), then repeats
  • Fields: event_id, user_id, amount, merchant, location, device, hour_of_day, risk_score_raw, anomaly_label, phase
  • Also publishes to phase-transitions on phase change
  • Metrics: events_produced_total, current_phase, anomaly_label_rate
raw-transactions
⚙️
Feature Engineer
:8002
Faust Streaming
Computes per-user sliding-window features using Redis. Publishes enriched events.

Feature Engineer

  • Reads raw-transactions, writes enriched-features
  • Redis keys: per-user amount history, tx count windows (1min, 5min, 1hr)
  • Features added: amount_zscore, tx_velocity_ratio, amount_vs_user_avg, is_high_risk_merchant, is_odd_hour, location_match, device_known
  • Faust agent with async processing
  • Metrics: events_enriched_total, enrichment_latency_seconds
enriched-features
🤖
ML Inference
:8003
River HalfSpaceTrees
Online anomaly scoring. Updates model on each event. Dynamic threshold. Persists to Redis.

ML Inference

  • Reads enriched-features, writes anomaly-scores
  • Ensemble: 70% HalfSpaceTrees + 30% StandardAbsoluteDeviation
  • Dynamic threshold: adjusts ±0.01 targeting 0.1%–5% anomaly rate
  • Model persistence: pickles to Redis key active_model every 1000 events
  • BatchWriter flushes scored_events → TimescaleDB every 100 rows or 5s
  • Metrics: events_processed_total, anomalies_detected_total, anomaly_score, current_threshold, model_version, rolling_anomaly_rate
anomaly-scores
📡
Drift Detector
:8004
River ADWIN + DDM
Detects data & concept drift. Fires retraining triggers.

Drift Detector

  • Reads anomaly-scores
  • ADWIN on: amount, amount_zscore, anomaly_score (data drift)
  • DDM on prediction error rate (concept drift)
  • Publishes drift events to drift-events + retraining-triggers
  • Writes to TimescaleDB drift_events table
  • Metrics: drift_events_total{type,feature}, time_since_last_drift_seconds
🚨
Alert Router
:8006
kafka-python
Severity bucketing, dedup via Redis TTL, Slack fanout.

Alert Router

  • Reads anomaly-scores, writes alerts
  • 4 severity levels: LOW / MEDIUM / HIGH / CRITICAL
  • Redis dedup TTLs: LOW=600s, MEDIUM=300s, HIGH=120s, CRITICAL=60s
  • Slack webhooks for severity ≥ MIN_SLACK_SEVERITY (default HIGH)
  • BatchWriter flushes alerts → TimescaleDB
  • Metrics: alerts_fired_total{severity}, alerts_deduplicated_total
retraining-triggers
🔁
Retraining Worker
:8005 / :5555
Celery + River
Trains new model on recent data. Shadow-tests vs incumbent. Promotes if better.

Retraining Worker

  • Reads retraining-triggers via Kafka consumer loop
  • Dispatches retrain_model Celery task (Redis broker)
  • Fetches last 50,000 rows from TimescaleDB scored_events
  • Trains new HalfSpaceTrees + SAD ensemble
  • Shadow-tests on last 200 events vs incumbent
  • Promotes if new precision ≥ old precision → writes to Redis active_model
  • Logs new version to model_versions table
  • Flower UI on :5555 for Celery task monitoring
  • Metrics: retraining_runs_total, retraining_duration_seconds, model_promotions_total
Kafka Topics (3 partitions each)
raw-transactions Raw synthetic financial events from Producer. Schema: event_id, user_id, amount, merchant, phase, anomaly_label, risk_score_raw.
enriched-features Feature-engineered events from Feature Engineer. Adds sliding-window statistics, Z-scores, velocity ratios.
anomaly-scores ML-scored events with anomaly_score, is_anomaly flag, threshold_used, model_version. Read by both Drift Detector and Alert Router.
drift-events Drift detection alerts with drift_type (data|concept), feature name, detector (ADWIN|DDM), timestamp.
retraining-triggers Signals from Drift Detector to trigger model retraining. Includes reason (data_drift|concept_drift) and urgency.
alerts Processed anomaly alerts after severity bucketing and deduplication. Consumed by downstream dashboards.
phase-transitions Emitted by Producer when simulation phase changes (Normal → Drift → Attack). Used for labelling and monitoring.
Phase Simulation Cycle
Normal Phase
Events 0 – 99,999
Baseline financial transactions. Low anomaly rate (~0.5%). Amount drawn from normal distribution (μ=150, σ=50). Model learns typical behaviour via online updates.
Drift Phase
Events 100,000 – 199,999
Distribution shift: higher amounts, different merchant mix. ADWIN detects shift in amount/amount_zscore/anomaly_score within ~1–2 min. Triggers retraining to adapt the model.
Attack Phase
Events 200,000 – 249,999
Simulated fraud burst. High-amount transactions, unusual hours, unknown devices. Anomaly rate spikes. DDM detects prediction error rate increase (concept drift). Cycle repeats.

Key Design Decisions
DecisionRationale
Online learning (River) HalfSpaceTrees updates on each event — no batch training cycle, no cold-start latency. Adapts continuously to distribution shifts.
Ensemble scoring (HST + SAD) 70% HalfSpaceTrees (tree-based anomaly isolation) + 30% StandardAbsoluteDeviation (statistical baseline). Blend reduces false positive rate.
Dynamic threshold Threshold adjusts ±0.01 per evaluation cycle, targeting 0.1%–5% anomaly rate. Prevents alert fatigue during drift phases without manual tuning.
Model persistence via Redis Pickled model stored in Redis active_model key every 1000 events. Survives container restarts. Retraining worker overwrites atomically on promotion.
Celery subprocess (no recursion) Retraining worker spawns celery -A main worker as a subprocess. Celery imports main as a module (not __main__), so the subprocess guard prevents infinite recursion.
BatchWriter pattern All TimescaleDB writes flush every 100 rows OR every 5s via a background thread. Batching reduces DB round-trips by ~100x vs. per-event inserts.
Shadow testing before promotion New model is compared against incumbent on the most recent 200 events. Only promoted if new precision ≥ old precision. Guards against degraded models from drift transients.
Redis dedup for alerts Per-severity TTLs (LOW=600s … CRITICAL=60s) prevent alert storms. Same event cannot fire the same severity again until TTL expires.
